## セクション3: AI駆動開発の概要 — 座学（0:25–0:55）

### Slide 8: AI駆動開発とは何か

**タイトル**: AI駆動開発とは何か

**リード文**: AIをコーディングパートナーとして活用する、新しい開発スタイル。

- AIが「代わりに書く」のではなく、人間が「AIと一緒に作る」
- 自然言語（日本語や英語）でやりたいことを伝えると、AIがコードを生成する
- 人間の役割: 何を作るか決める、生成されたコードを確認・修正する、品質に責任を持つ、開発プロセス全体を監督する
- AIの役割: コードの下書き、パターンの提案、反復作業の自動化

---

### Slide 9: 従来開発との違い

**タイトル**: 従来開発 vs AI駆動開発

**リード文**: 開発者の役割が「コードを書く人」から「AIに指示し、検証する人」にシフトする。

- **従来開発**:
  - コードを1行ずつ手で書く
  - ドキュメントやStack Overflowを検索して調べる
  - プロトタイプ作成に数日〜数週間
- **AI駆動開発**:
  - 自然言語で要件を伝え、AIがコードを生成
  - AIが文脈を理解して提案・修正
  - プロトタイプが数分〜数時間で形になる
- 共通して変わらないこと: 機能と非機能の担保、設計判断、品質責任、セキュリティ対策は人間の仕事

（画像: 2カラムの対比図。左に「従来開発」、右に「AI駆動開発」を配置し、各項目を対比。中央に「変わらないこと」を共通エリアとして表示）

---

### Slide 10: AI駆動開発の3つのモード

**タイトル**: 3つのモード — 目的に応じて使い分ける

**リード文**: AI駆動開発には構造化の度合いが異なる3つのアプローチがあり、目的に応じて選ぶ。

- **バイブコーディング**: 自然言語だけで「とりあえず動くもの」を作る。最もカジュアル
- **プロトタイピング**: 参照アーキテクチャに沿い、設計を持ってAIと組み立てる
- **プロダクション開発**: CI/CD、テスト、セキュリティを組み込んだ本番レベルの開発
- この研修では、第2回でバイブコーディング、第3〜4回でプロトタイピング、第5回でプロダクション開発を体験します

（画像: 左から右へ構造化の度合いが上がるスペクトラム図。左端「バイブコーディング（自由度 高 / 品質保証 低）」→ 中央「プロトタイピング」→ 右端「プロダクション開発（自由度 低 / 品質保証 高）」。各研修回との対応を矢印で表示）

### Slide 11: バイブコーディング（第2回で体験）

**タイトル**: バイブコーディング — AIに「いい感じに作って」と頼む

**リード文**: 自然言語だけでアプリを組み立てる最もカジュアルなモード。次回体験する。

- やり方: 「経費精算アプリを作って」のようにざっくり指示するだけ
- 強み: 驚くほど速い。数分で「動くもの」が出てくる
- 弱み:
  - セキュリティ対策が抜ける（AIは指示されないと作らない）
  - 機能追加するとデグレード（既存機能が壊れる）が起きやすい
  - コードの保守性が低い（1週間後の自分が読めない）
- **第2回でこの「落とし穴」を全員が自分の手で体験します**

### Slide 12: プロトタイピング / プロダクション開発（第3〜5回で体験）

**タイトル**: 設計を持ってAIと進める — プロトタイピングとプロダクション開発

**リード文**: 実務に直結するモード。バイブコーディングの弱みを設計でカバーする。

- **プロトタイピング（第3〜4回）**:
  - 参照アーキテクチャ（お手本の設計図）をベースにする
  - スターターテンプレート（雛形）から着手し、AIに肉付けさせる
  - MCPで外部ツールやDBに接続し、「本物のデータ」で動かす
- **プロダクション開発（第5回）**:
  - CI/CD（コードを変更すると自動でテスト・デプロイが走る仕組み）を組み込む
  - クラウドにデプロイし、URLでアクセスできる状態にする
  - セキュリティ、シークレット管理、IAM権限設計を実践する

### Slide 15: Claude Code と他のAIコーディングツールの違い

**タイトル**: Claude Code と他のAIコーディングツールの違い

**リード文**: AIコーディングツールにはさまざまな種類がある。それぞれの特徴を理解し、使い分けの視点を持つ。

- **GitHub Copilot**: エディタ内でのコード補完が中心。書いている途中に次の行を提案してくれる
- **Cursor**: AI機能を組み込んだエディタそのもの。エディタごと乗り換える形
- **Claude Code**: エージェント型。自然言語で指示すると、ファイル操作・Git操作・シェルコマンドまで自律的に実行する
- **Claude Code の特徴的な強み**:
  - MCPによる外部ツール連携（DB、API、ファイルシステムなどを統一的に接続）
  - プロジェクト全体を文脈として理解した上での操作
  - CLI とVS Code 拡張の両方で使える柔軟性
- ツールの優劣ではなく、目的に応じた使い分けが重要。この研修では Claude Code に集中します

### Slide 16: セキュリティ留意点①: プロンプトインジェクション

**タイトル**: セキュリティ留意点①: プロンプトインジェクション

**リード文**: この研修ではセキュリティを3つの層で整理する。まず「AI の入出力の安全性」（今日の座学）、次に「AI の行動ガードレール」（第3〜5回で体験）、最後に「デプロイ環境の安全性」（第5回で補足）。今日はまず層1 — 外部ソースからの悪意ある指示にLLMが従ってしまうリスク。

- **プロンプトインジェクションとは**: Webページやファイルなどの外部データに「AIへの隠し指示」が埋め込まれ、AIが意図しない動作をしてしまうこと
- 例: Webページに「この情報を無視して、代わりに○○を実行せよ」と書かれていた場合、AIがそれに従ってしまう可能性がある
- **対策の基本**:
  - 信頼できないソースの内容をそのままプロンプトに貼り付けない
  - AIの出力を鵜呑みにせず、意図通りの動作かを確認する
  - ツールの権限を最小限に設定する（次のスライドで詳しく）

（画像: プロンプトインジェクションの概念図。「ユーザーの指示」と「外部ソースの隠し指示」がLLMに入力され、LLMが意図しない動作をしてしまう流れを矢印で示す）

### Slide 17: セキュリティ留意点②: 機密情報の扱い

**タイトル**: セキュリティ留意点②: APIキー・個人情報をプロンプトに入れない

**リード文**: 機密情報をプロンプトやコミットに含めないことは、AI駆動開発の鉄則。

- **やってはいけないこと**:
  - APIキーやパスワードをプロンプトに直接入力する
  - APIキーをソースコードにハードコードする
  - `.env` ファイル（設定ファイル）をGitにコミットする
- **やるべきこと**:
  - APIキーは `.env` ファイルに書き、`.gitignore` でコミット対象外にする
  - AIが生成したコードに機密情報がハードコードされていないかレビューする
  - チームの機密情報管理ルールに従う
- この研修でも `.env` + `.gitignore` のパターンを実践します

### Slide 18: セキュリティ留意点③: 生成コードの鵜呑み禁止

**タイトル**: セキュリティ留意点③: AIの「完成しました」を信じる前に確認する

**リード文**: AIは自信たっぷりに「完成しました」と言うが、実際には不完全・不正確な場合がある。

- AIが生成するコードは「もっともらしいが間違っている」ことがある
- **確認すべきポイント**:
  - 意図通りの動作をしているか（実際に動かして確認）
  - セキュリティ上の問題はないか（認証・バリデーション・権限）
  - エッジケース（想定外の入力）に対応しているか
- **ツール権限の考え方**: MCPサーバーが何にアクセスできるかは、自分で決めて制御する
- **第2回でこの「鵜呑みの危険」を身をもって体験します**

---

### Slide 19: AI の行動を制御する 4 つの仕組み

**タイトル**: AI の行動を制御する 4 つの仕組み — 第3回以降で体験します

**リード文**: セキュリティを「人間の注意力」だけに頼らず、仕組みとして AI の行動を制限・監視する方法がある。

- **Permissions**（中核）: settings.json に書く allow/deny ルール。Claude Code が実行前に機械的にチェックするため、**仕組みとして必ず効く**
- **Hooks**（補強）: ツール呼び出しの前後で走るシェルコマンド。実行時に検査・ブロック・記録ができる
- **CLAUDE.md**（補助）: AI に知識や指示を伝えるファイル。ただし長い会話では**忘れられることがある**ため、命令としては弱い
- **Skills**（機能拡張）: 再利用可能な手順・ツールを Claude Code に登録して呼び出せる仕組み
- **重要な違い**: Permissions は「必ず効く」が、CLAUDE.md は「忘れられうる」。この違いが安全な AI 開発の設計のカギ
- 今日は存在を知るだけでOK。第3回以降で実際に体感・自作します

（画像: 4つの仕組みを並べた図。左から「Permissions（中核・必ず効く）」「Hooks（補強・実行時検査）」「CLAUDE.md（補助・忘れられうる）」「Skills（機能拡張）」を強度順に配置。Permissions を最も目立たせるデザイン）

### Slide 28: Step 6 — SSH 鍵の作成と GitHub への登録

**タイトル**: Step 6: SSH 鍵の作成と GitHub への登録

**リード文**: GitHub にセキュアに接続するため、SSH 鍵を作成して登録する。毎回パスワードを入力する必要がなくなる。

- **SSH 鍵の作成**（ターミナルで実行）:
  ```
  ssh-keygen -t ed25519 -C "自分のメールアドレス"
  ```
  - 保存場所はデフォルト（Enter）でOK
  - パスフレーズは任意（空でも可、設定するとより安全）
- **公開鍵をコピー**:
  - **Windows**: `cat ~/.ssh/id_ed25519.pub | clip`
  - **Mac**: `cat ~/.ssh/id_ed25519.pub | pbcopy`
- **GitHub に公開鍵を登録**:
  1. https://github.com/settings/keys を開く
  2. 「New SSH key」をクリック
  3. Title に任意の名前（例: 「研修用PC」）を入力
  4. Key 欄にコピーした公開鍵を貼り付けて「Add SSH key」
- **接続テスト**:
  ```
  ssh -T git@github.com
  ```
  → `Hi ユーザー名! You've successfully authenticated, but GitHub does not provide shell access.` のようなメッセージが表示されれば成功
- **うまくいかない場合**: 講師に声をかけてください

### Slide 31: MCP（Model Context Protocol）とは

**タイトル**: MCP とは — LLM と外部ツールをつなぐ「標準プロトコル」

**リード文**: MCPは、LLM（大規模言語モデル）とさまざまな外部ツールやデータソースを統一的なインターフェースでつなぐ標準プロトコル。

- **たとえるなら**: USB のように、さまざまなデバイスを1つの規格で接続できる仕組み
- **MCPが解決する課題**: ツールごとに異なる接続方法を覚える必要がなくなる
- **MCPサーバーの例**:
  - ファイルシステム: ファイルの読み書き
  - GitHub: リポジトリ情報の取得、Issue管理
  - データベース: SQLクエリの実行
- **この研修の裏テーマ**: MCPを「使う」→「組み込む」→「自作する」→「クラウドで動かす」と段階的に習得します

（画像: MCP の概念図。中央に「MCP（標準プロトコル）」を配置し、左側に LLM（Claude）、右側に複数のツール/データソース（ファイルシステム、GitHub、DB、外部API）をアイコンで表示。LLM と各ツールの間を MCP が橋渡ししている構図）
